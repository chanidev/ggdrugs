## 1. 추진 배경 및 문제 정의

### 1-1. 문제 의식

국내 이벤트·행사 정보는 **공급원이 극도로 분산**되어 있다. 동일한 행사 정보가 한국관광공사 TourAPI, 서울 열린데이터광장, 한국문화정보원(KCISA) 등 복수의 공공 API에 산재하며, 각 소스의 분류 체계·지역 표기·기간 표기가 서로 달라 사용자가 한곳에서 통합 탐색하기 어렵다. 또한 기존 검색은 **키워드 매칭 중심**이어서 "이번 주말 연인이랑 강남에서 볼만한 공연" 같은 **자연어 의도**를 그대로 받아내지 못한다.

### 1-2. 세 가지 미충족 니즈

| 니즈 | 현황의 한계 | Alle의 접근 |
|---|---|---|
| **자연어 이벤트 검색** | 키워드·필터 조합을 사용자가 직접 구성해야 함 | 대화형 채팅으로 의도를 5종 필터 + 의미 검색으로 자동 변환 |
| **분산된 행사 정보 통합** | 소스마다 포맷·지역·분류가 상이 | 3개 공공 API를 단일 스키마로 정규화·중복제거·지오코딩 |
| **동행(메이트) 매칭 니즈** | 같은 행사를 함께 갈 동행을 찾을 채널 부재 | 같은 이벤트 선택자 풀에서 규칙 기반 양방향 매칭 + 채팅·약속·평가 |

### 1-3. 데이터의 시의성·이슈 연계

이벤트는 **시점성**이 강하다. 종료된 행사를 추천하면 가치가 없으므로 `phase`(upcoming/ongoing/ended) 상태머신과 기간 필터로 시점을 통제한다. 나아가 각 이벤트를 **뉴스 기사와 매핑**하여(이벤트·이슈 연계) 사용자가 행사의 맥락·화제성까지 함께 파악하도록 한다. 현재 approved 이벤트 4,111건 중 1,810건(44%)에 뉴스 매핑이 연결되어 있다.

## 2. 서비스 소개 및 핵심 가치

### 2-1. 한 줄 정의

> 공공 API에서 이벤트를 수집하여 AI로 요약·임베딩·뉴스매핑한 뒤, **지도 + 자연어 채팅 UI**에서 검색·추천·구독하고, 같은 행사를 갈 **동행(메이트)을 매칭**해 주는 자연어 기반 이벤트·이슈 지도 검색 및 동행 매칭 서비스.

### 2-2. 사용자 역할 3종 ("1계정 = 복수 역할 토글")

별도 계정을 만들지 않고, 하나의 계정에 역할을 토글로 추가하는 모델이다. 현재 활성 역할은 `active_role`로 관리한다.

| 역할(role) | 핵심 행위 | 비고 |
|---|---|---|
| **일반 사용자(user)** | 이벤트 탐색·북마크·리뷰 작성·구독·추천 수신, (Phase 2) 커뮤니티·메이트 동행 매칭 | 기본 역할 |
| **업로더(uploader)** | 이벤트 등록 (축제 기획자·사설 단체·공공기관). 등록 건은 심사 대상 | 일반 사용자에 역할 추가 |
| **관리자(admin)** | 콘텐츠 승인, event_vibe 라벨 부여, 업로더 승급 심사, 신고 모더레이션 | scope 4종(full/content_only/uploader_review_only/security) |

### 2-3. 핵심 가치 4축

1. **자연어 채팅 검색** — 대화형으로 의도를 입력하면 필터 5종 + 의미 검색 + 재정렬을 거쳐 추천을 반환(SSE 스트리밍).
2. **지도 중심 탐색** — Kakao Maps 위에서 핀·클러스터·뷰포트 기반으로 행사를 직관적으로 탐색.
3. **이벤트-이슈 연계** — 이벤트마다 관련 뉴스 기사를 매핑하여 맥락·화제성 제공.
4. **동행 매칭(Phase 2)** — 같은 행사 선택자 풀에서 규칙 기반 매칭 → 1:1/그룹 채팅 → 약속 합의 → 평가 → 크레딧 적립.

### 2-4. 이벤트 상태 머신

```
pending → revision_requested → pending  (재제출)
pending → approved            → ended   (종료일 도래)
pending → rejected                       (종결)
```

### 2-5. 필터 5종 (고정)

**지역 / 기간 / 인원구성 / 이벤트 종류 / 이벤트 성향** — 이 5종 외 필터 추가는 요구사항정의서 개정이 선행되어야 한다. (메이트 매칭의 선호조건은 *별개 도메인*으로, 이벤트 필터에 추가하지 않는다.)

## 4. 시장·필요성 관점 (STP)

견본의 "시장 분석 / STP / 필요성" 섹션을 채울 재료다.

### 4-1. Segmentation (세분화)

| 세그먼트 | 특성 | 핵심 페인 |
|---|---|---|
| 행사 탐색 일반 사용자 | 주말·여가 활동을 자연어로 찾고 싶은 사용자 | 분산된 정보, 키워드 검색의 한계 |
| 동행 탐색 사용자(Phase 2) | 혼자 가기 부담되는 행사에 함께 갈 사람 필요 | 신뢰 가능한 동행 채널 부재 |
| 외국인·다국어 사용자(Phase 2) | 국내 행사 정보 접근 장벽 | 언어 장벽 (UI·게시글) |
| 업로더(기획자·단체·공공기관) | 행사 노출 채널 확보 | 통합 노출 플랫폼 부재 |

### 4-2. Targeting (표적)

1차 표적은 **수도권·전국 단위 여가 행사를 자연어로 탐색하는 일반 사용자**다. ADR 0006으로 서울 전용에서 **전국 17개 시/도 + 약 230개 시/군/구**로 지역을 확장하여 비-서울 사용자 유입 기반을 마련했다. 2차 표적은 동행 매칭을 원하는 사용자와 다국어(한/영/베트남/중국/일본/프랑스) 사용자다.

### 4-3. Positioning (포지셔닝)

> "키워드 목록이 아니라 **대화로 찾고, 지도에서 보고, 함께 갈 사람까지 매칭**되는 이벤트 서비스."

| 차별점 | 설명 |
|---|---|
| 자연어 채팅 검색 | 5종 필터 + 의미 검색 + LLM 재정렬(matchReason 1줄) 결합 |
| 지도 우선 하이브리드 UI | map-first 레이아웃, 핀 클릭 시 요약 패널 |
| 이벤트-이슈 연계 | 행사에 관련 뉴스 매핑(44% 커버리지) |
| 동행 매칭 + 메이트 지수 | 규칙 기반 투명 매칭 + 평가 기반 신뢰지표("당근 온도"형) |
| 설명가능성 | 매칭·필터를 결정적으로 계산 → LLM에 관리자/매칭 판단을 위임하지 않음 |

### 4-4. 필요성 요약

분산 정보 통합(공급), 자연어 의도 수용(검색 UX), 동행이라는 사회적 니즈(리텐션) — 세 축이 동시에 충족되는 통합 서비스가 시장에 부재하다는 점이 본 프로젝트의 추진 근거다.

## 5. 빅데이터 분석 관점

견본의 "빅데이터 분석 정의서"를 채울 재료다.

### 5-1. 데이터 수집원 (Sources)

| 소스 | 수집 대상 | 비중 |
|---|---|---|
| Seoul Open Data | 서울 문화행사 8 카테고리 | 압도적 다수 |
| TourAPI(한국관광공사) | 전국 축제(forward-looking) | 전국 확장 핵심 |
| KCISA(한국문화정보원) | 공연·전시 | 키 의존(보유 상태에 따라 변동) |
| Naver 뉴스 검색 API + Google News RSS | 이벤트-뉴스 매핑 | 뉴스 레이어 |

**현재 적재 규모**: approved 이벤트 **4,111건**, phase 분포 upcoming 163 / ongoing 260 / ended 3,661. 뉴스 매핑 **1,810건 = 44% 커버리지**.

### 5-2. 수집·정제 파이프라인 (24h 주기 scheduler)

```
부팅+2s, 24h 주기 — 3 러너 병렬 수집(TourAPI/Seoul/KCISA)
└ ingest-common 8단계
   1) 지역 guard  2) 시군구 추출  3) regionId resolve
   4) 카테고리 8종 매핑  5) cross-source 중복제거(title+start_date)
   6) phase 계산  7) approval_status 부여  8) Prisma upsert
└ 후속 직렬 4단계
   1) AI 요약(gpt-4o-mini)  2) 뉴스 매핑(Naver+Google)
   3) 임베딩 → Qdrant upsert  4) 매핑 분포 감사
```

지오코딩·지역 정규화는 `(sido, sigungu)` exact → 시 단위 → 광역 → null 순의 fallback resolver로 처리하며, ADR 0006 실측에서 TourAPI 표본 393건 전수 매칭(0 errors)을 확인했다.

### 5-3. AI 분석·임베딩 파이프라인

| 분석 작업 | 방법 | 산출 컬럼 |
|---|---|---|
| 이벤트 요약 | gpt-4o-mini, "신문 도입부 톤, 2~3문장, 250자, 사실만" 프롬프트 | `events.ai_summary` |
| 임베딩 | text-embedding-3-small (1536d) → Qdrant `alle-events` | Qdrant 포인트 |
| 뉴스 매핑 점수 | `final = 0.4×keyword_overlap + 0.6×embedding_cosine` (임계 0.60/0.55) | `event_article_mappings.relevance_score` |
| 리뷰 감성분석 | gpt-4o-mini 분류(positive/negative/neutral), 키 없으면 사전 기반 | `reviews.sentiment` |
| 사진 태깅 | AI 태깅 | `photos.ai_tags` (JSONB) |

### 5-4. LLM 활용 (검색·요약·추천 — 판단 위임 금지)

- **자연어 채팅 검색(/chat/stream 5-step)**: 개인화 컨텍스트(taste profile + 최근 북마크) → LLM 필터 5종 + reply + followups 추출 → 지역/성향 ID resolve → Qdrant 의미 검색(over-fetch 30, score≥0.25) → 후보≥6 & 8자↑ 시 LLM 재정렬(top 12→top 5 + matchReason) → 0건 시 retreat 안내.
- **추천(/me/recommendations)**: `user_taste_profiles` 3차원(선호 카테고리/지역/성향) + seed 이벤트 mean-vector kNN.
- **게시글 번역(Phase 2)**: 온디맨드 6개 언어 번역 + 결과 캐시.

### 5-5. 모델·파인튜닝 여부

**파인튜닝(fine-tuning) 없음.** Alle의 LLM 활용은 **사전학습 모델(gpt-4o / gpt-4o-mini) + structured outputs 프롬프팅 + 임베딩 기반 RAG**(text-embedding-3-small → Qdrant 검색 → 후보를 LLM이 재정렬·요약) 구조다. 자체 모델 학습이나 가중치 갱신은 수행하지 않으며, 비용·예산 한도 초과 또는 키 미설정 시 규칙 기반으로 조용히 fallback한다. (CLAUDE.md 금지 #4 — 이벤트 승인·라벨·승급 등 관리자 판단은 LLM에 위임하지 않음.)

### 5-6. Phase 2 매칭 분석 (보강)

메이트 동행 매칭은 **벡터/임베딩이 아닌 규칙 기반 양방향 가중 점수**로 결정적 계산한다(ADR 0007): 하드 필터(불일치 제외) → 소프트 점수(지역 근접·연령대 차·국적/한국어/자차 가중 합산) → 동점은 **메이트 지수**(평가 기반 집계, 사용자 수정 불가)로 정렬. 설명가능성·디버깅 용이성을 위해 LLM을 의도적으로 배제한다.

### 6-2. 범위 (Scope)

**In-scope**: 3개 공공 API 수집·정규화, AI 요약·임베딩·뉴스매핑, 자연어 채팅 검색, Kakao 지도 UI, 필터 5종, 역할 3종·승인 흐름, 구독·알림·추천, (Phase 2) 커뮤니티·메이트·크레딧·다국어·캘린더.
**Out-of-scope**: AI 비디오 생성(v5.0에서 제거), 크레딧 실지불 지갑/PG 연동(향후 별도 ADR), LLM의 관리자 판단 위임.

### 6-3. 협업 인원 역할표

CLAUDE.md의 에이전트 역할 분담을 인원 역할표 재료로 정리한다.

| 역할 | 담당 디렉터리 | 주요 책임 |
|---|---|---|
| **Orchestrator** | `docs/` | 전체 조율, 문서·ADR 관리, 에이전트 간 의존성 충돌 조정 |
| **Backend** | `apps/bff/`, `packages/shared-types/` | BFF API, Prisma 스키마, 인증, 권한 관리 |
| **Frontend** | `apps/web/` | React 컴포넌트, Kakao Maps 통합, 상태 관리, 채팅 UI |
| **LLM** | `services/llm/` | 필터 추출·재정렬·요약·임베딩 체인, 프롬프트, Qdrant 연동 |
| **Infra** | `infra/` | Docker 이미지, Compose, 마이그레이션, 로컬 환경 안정화 |
| **QA** | (교차) | 테스트 작성, 유스케이스 기반 시나리오 검증, DESIGN.md 편차 flag |

### 6-4. 협업 규약 (요약)

- 용어 통일(event는 절대 festival로 대체 금지 등), 필터 5종 고정, Conventional Commits, 브랜치 전략(main/dev/feature·fix·chore).
- DB 변경은 Prisma 마이그레이션으로만, 환경변수는 `.env.example` + `packages/config/schema.ts` 동시 갱신.
- PII(주민번호·전화·이메일) 로그 출력 금지·마스킹, UI/시각 결정 전 DESIGN.md 선독.
