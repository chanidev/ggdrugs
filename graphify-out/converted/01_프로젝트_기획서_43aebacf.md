# 프로젝트 기획서
과제명: 자연어 기반 이벤트·이슈 지도 검색 및 동행 매칭 서비스 — Alle
작성일: 2026.06.02
작성: Alle 팀

## 변경이력

## 1. 프로젝트 소개
Alle는 전국에 분산된 이벤트·행사 정보를 한곳에 모아, 지도와 자연어 채팅으로 탐색하고, 같은 행사를 함께 갈 동행(메이트)을 매칭해 주는 서비스다.
공공 API(한국관광공사 TourAPI, 서울 열린데이터광장, 한국문화정보원 KCISA)에서 이벤트를 수집하여 단일 스키마로 정규화한 뒤, AI로 요약·임베딩·뉴스매핑을 수행한다. 사용자는 키워드 목록이 아니라 "이번 주말 연인이랑 강남에서 볼만한 공연" 같은 자연어 문장으로 의도를 입력하고, Kakao Maps 위에서 핀·클러스터로 행사를 직관적으로 탐색하며, 각 행사에 연결된 관련 뉴스로 맥락·화제성까지 함께 파악한다.
한 줄 정의 공공 API에서 이벤트를 수집하여 AI로 요약·임베딩·뉴스매핑한 뒤, 지도 + 자연어 채팅 UI에서 검색·추천·구독하고, 같은 행사를 갈 동행(메이트)을 매칭해 주는 자연어 기반 이벤트·이슈 지도 검색 및 동행 매칭 서비스.
### 1-1. 사용자 역할 3종 — "1계정 = 복수 역할 토글"
별도 계정을 만들지 않고, 하나의 계정에 역할을 토글로 추가하는 모델이다. 현재 활성 역할은 active_role로 관리한다.
### 1-2. 핵심 가치 4축
자연어 채팅 검색 — 대화형으로 의도를 입력하면 필터 5종 + 의미 검색 + 재정렬을 거쳐 추천을 반환(SSE 스트리밍).
지도 중심 탐색 — Kakao Maps 위에서 핀·클러스터·뷰포트 기반으로 행사를 직관적으로 탐색.
이벤트-이슈 연계 — 이벤트마다 관련 뉴스 기사를 매핑하여 맥락·화제성 제공.
동행 매칭(Phase 2) — 같은 행사 선택자 풀에서 규칙 기반 매칭 → 1:1/그룹 채팅 → 약속 합의 → 평가 → 크레딧 적립.

## 2. 추진 배경
### 2-1. 문제 의식
국내 이벤트·행사 정보는 공급원이 극도로 분산되어 있다. 동일한 행사 정보가 한국관광공사 TourAPI, 서울 열린데이터광장, 한국문화정보원(KCISA) 등 복수의 공공 API에 산재하며, 각 소스의 분류 체계·지역 표기·기간 표기가 서로 달라 사용자가 한곳에서 통합 탐색하기 어렵다. 또한 기존 검색은 키워드 매칭 중심이어서 "이번 주말 연인이랑 강남에서 볼만한 공연" 같은 자연어 의도를 그대로 받아내지 못한다.
### 2-2. 세 가지 미충족 니즈
### 2-3. 데이터의 시의성·이슈 연계
이벤트는 시점성이 강하다. 종료된 행사를 추천하면 가치가 없으므로 phase(upcoming / ongoing / ended) 상태와 기간 필터로 시점을 통제한다. 나아가 각 이벤트를 뉴스 기사와 매핑하여(이벤트·이슈 연계) 사용자가 행사의 맥락·화제성까지 함께 파악하도록 한다. 현재 approved 이벤트 4,111건 중 1,810건(44%)에 뉴스 매핑이 연결되어 있다.

## 4. 프로젝트 내용 (제안 내용)
### 4-1. 개발 배경
분산된 이벤트 정보를 사용자가 직접 여러 채널에서 찾아 비교해야 하는 현재의 탐색 경험은 비효율적이며, 키워드 검색은 사용자의 자연어 의도를 수용하지 못한다. 동시에 "같은 행사를 함께 갈 사람을 찾고 싶다"는 사회적 니즈를 충족하는 신뢰 가능한 채널이 부재하다. Alle는 (1) 공급 측 분산 통합, (2) 검색 UX의 자연어화, (3) 동행이라는 사회적 니즈를 하나의 서비스로 묶어 해결한다.
### 4-2. 개발 내용
### 4-3. 제공 기능 — 이벤트 상태 머신
pending → revision_requested → pending  (재제출)
pending → approved            → ended   (종료일 도래)
pending → rejected                       (종결)
approval_status(심사 상태: pending/revision_requested/approved/rejected)와 phase(시점 상태: upcoming/ongoing/ended)는 별개의 2축이다.
### 4-4. 제공 기능 — 필터 5종 (고정)
지역 / 기간 / 인원구성 / 이벤트 종류 / 이벤트 성향 — 이 5종 외 필터 추가는 요구사항정의서 개정이 선행되어야 한다.
메이트 매칭의 선호조건은 별개 도메인으로, 이벤트 필터에 추가하지 않는다.

## 5. 시장 분석
### 5-1. 시장 규모·동향
여가·문화 소비와 지역 축제 관광 수요는 코로나 회복 이후 꾸준히 확대되는 추세다(추정). 국내 지역축제는 문화체육관광부가 매년 「지역축제 개최계획」으로 집계·공개하며 연간 1,000건 이상 규모로 개최된다(출처: 문화체육관광부 지역축제 개최계획, 연도별 공개). 동시에 공공데이터 개방 정책으로 행사 정보가 다수의 공공 API로 제공되고 있으나, 이를 사용자 친화적으로 통합·탐색하는 서비스는 부족하다. 한편 자연어 인터페이스(LLM 기반 검색·추천)와 '함께하기'형 소셜 매칭은 최근 수년간 대중적 수용도가 빠르게 높아진 영역으로(추정), Alle는 이 두 흐름의 교차점에 위치한다.
외부 시장 규모의 구체 수치는 본 프로젝트 산출물 범위에서 확정된 근거가 없어 (TBD)로 표기하며, 위 동향 서술은 일반적 추세에 근거한 (추정)이다.
### 5-2. 기존 서비스 비교
### 5-3. STP 전략
① Segmentation (세분화)
② Targeting (표적)
1차 표적은 수도권·전국 단위 여가 행사를 자연어로 탐색하는 일반 사용자다. 서울 전용에서 전국 17개 시/도 + 약 230개 시/군/구로 지역을 확장하여 비-서울 사용자 유입 기반을 마련했다. 2차 표적은 동행 매칭을 원하는 사용자와 다국어(한/영/베트남/중국/일본/프랑스) 사용자다.
③ Positioning (포지셔닝)
"키워드 목록이 아니라 대화로 찾고, 지도에서 보고, 함께 갈 사람까지 매칭되는 이벤트 서비스."
### 5-4. 필요성
분산 정보 통합(공급), 자연어 의도 수용(검색 UX), 동행이라는 사회적 니즈(리텐션) — 세 축이 동시에 충족되는 통합 서비스가 시장에 부재하다는 점이 본 프로젝트의 추진 근거다.

## 7. 개발 내용 (기능 상세)
### 7-1. 데이터 수집원 (Sources)
현재 적재 규모: approved 이벤트 4,111건, phase 분포 upcoming 163 / ongoing 260 / ended 3,661. 뉴스 매핑 1,810건 = 44% 커버리지.
### 7-2. 수집·정제 파이프라인 (24h 주기 scheduler)
부팅+2s, 24h 주기 — 3 러너 병렬 수집(TourAPI / Seoul / KCISA)
└ ingest-common 8단계
   1) 지역 guard   2) 시군구 추출   3) regionId resolve
   4) 카테고리 8종 매핑   5) cross-source 중복제거(title+start_date)
   6) phase 계산   7) approval_status 부여   8) Prisma upsert
└ 후속 직렬 4단계
   1) AI 요약(gpt-4o-mini)   2) 뉴스 매핑(Naver+Google)
   3) 임베딩 → Qdrant upsert   4) 매핑 분포 감사
지오코딩·지역 정규화는 (sido, sigungu) exact → 시 단위 → 광역 → null 순의 fallback resolver로 처리하며, 전국 확장 검증에서 TourAPI 표본 393건 전수 매칭(0 errors)을 확인했다.
### 7-3. AI 분석·임베딩 파이프라인
### 7-4. 자연어 채팅 검색 (/chat/stream 5-step)
개인화 컨텍스트(taste profile + 최근 북마크) → LLM 필터 5종 + reply + followups 추출 → 지역/성향 ID resolve → Qdrant 의미 검색(over-fetch 30, score ≥ 0.25) → 후보 ≥ 6 & 8자↑ 시 LLM 재정렬(top 12 → top 5 + matchReason) → 0건 시 검색 무응답(결과 0건) 안내.
### 7-5. 추천 (/me/recommendations)
user_taste_profiles 3차원(선호 카테고리 / 지역 / 성향) + seed 이벤트 mean-vector kNN.
### 7-6. 모델·파인튜닝 여부
파인튜닝(fine-tuning) 없음. Alle의 LLM 활용은 사전학습 모델(gpt-4o / gpt-4o-mini) + structured outputs 프롬프팅 + 임베딩 기반 RAG(text-embedding-3-small → Qdrant 검색 → 후보를 LLM이 재정렬·요약) 구조다. 자체 모델 학습이나 가중치 갱신은 수행하지 않으며, 예산 한도 초과 또는 키 미설정 시 규칙 기반으로 조용히 fallback한다. 이벤트 승인·라벨·승급 등 관리자 판단은 LLM에 위임하지 않는다.
### 7-7. (Phase 2) 메이트 동행 매칭
메이트 동행 매칭은 벡터/임베딩이 아닌 규칙 기반 양방향 가중 점수로 결정적 계산한다: 하드 필터(불일치 제외) → 소프트 점수(지역 근접·연령대 차·국적/한국어/자차 가중 합산) → 동점은 메이트 지수(평가 기반 집계, 사용자 수정 불가)로 정렬. 설명가능성·디버깅 용이성을 위해 LLM을 의도적으로 배제한다.
