# ADR 0007: Phase 2 — 커뮤니티 + 메이트(동행 매칭) 모듈

- **Status**: Proposed (2026-05-29) — Orchestrator 승인 대기 (CLAUDE.md 금지 #1)
- **Context**: Phase 1(이벤트 검색 · 업로더 · 관리자)은 가동 중. 그 위에 **커뮤니티(게시판) + 메이트 동행 매칭(1:1/그룹 채팅 → 약속 → 평가 → 크레딧)** 소셜 레이어를 신규 추가한다. 근거 문서:
  - `llm_wiki/raw/페이즈 2 요구사항정의서.pdf` (40p, 2026-04-27) — 신규 유스케이스 15종(A_800~A_901, A_701 등), 기능요구사항 ~120건(GG-COMM / MATCH / MATE / ROOM / POST / NOTI / PROFILE / REVIEW / FEST-REVIEW / REPORT), 비기능 ~20건.
  - 와이어프레임 23종 (`llm_wiki/raw/전체 와이어 프레임.zip`).
  - 흐름도 3종 (`llm_wiki/raw/페이즈 2 흐름도.zip`): 메이트 매칭 흐름 · 메인 유저 플로우 · 간략 흐름도.
- **Decision drivers**: 사용자 리텐션·소셜 확장, 기존 인프라(Redis/OpenAI/Postgres/Notification) 재사용, CLAUDE.md 컨벤션(용어 통일·금지사항)과의 정합, 명세에 미정의된 설계 공백 해소.

## Decisions

### 스코프

1. **전체 구현.** Phase 2 정의서를 정식 요구로 전부 구현한다. 와이어프레임의 "기능X" / 흐름도의 "기능x" 표기는 **무시**(전부 구현 대상)한다. (사용자 확정 2026-05-29). 단 일부 화면의 placeholder는 결정 5 참조.
2. **신규 도메인 = 커뮤니티(게시판) + 메이트(동행 매칭).** 대상 액터는 **일반 사용자**. 업로더·관리자·이벤트 검색(Phase 1)은 불변. 진입은 마이페이지 → 커뮤니티(A_800).

### 설계 결정 (명세 공백을 메우는 6대 결정 + 보강)

3. **메이트 매칭 알고리즘 = 규칙 기반 양방향 가중 점수** (벡터/임베딩 X).
   - 후보 풀 = *같은 축제(2주 이내 개최) 선택 + 매칭 동의* 한 사용자.
   - 양방향 평가: `내 선호조건 ↔ 상대 프로필` + `상대 선호조건 ↔ 내 프로필`.
   - **하드 필터**: '상관없음' 아닌 선호조건 불일치 → 후보 제외.
   - **소프트 점수**: 지역 근접 · 연령대 차 · 국적 / 한국어 / 자차 가중 합산. 동점은 **메이트 지수**로 정렬.
   - BFF/Postgres에서 결정적으로 계산. **LLM 미사용** (CLAUDE.md 금지 #4 준수). 흐름도의 "매칭 프로필 계산 → 추천목록 → 필터/정렬"에 대응 (GG-MATCH-012).

4. **메이트 지수("당근 온도") = 평가 기반 결정적 집계, 사용자 수정 불가** (GG-PROFILE-005).
   - 기준점에서 시작, 메이트 평가(4문항 ×1~5 + 별점)로 가중 이동평균 갱신, 최근값 가중. 신고/제재 시 감점.
   - 원본 평가(MateEvaluation) + 파생 지수 컬럼을 분리 저장.

5. **크레딧 = 적립형 리워드 카운터** (사용자 확정: 옵션 C).
   - 적립 소스 = 흐름도의 "메이트 활동 완료/평가 작성 후 지급"만. 9-1 크레딧 내역 화면의 **결제(−) · 멤버십 정산 · 구매적립 · 쿠폰적립 항목은 placeholder로 간주** — 결제/PG 연동 **없음**.
   - **용어 "크레딧"으로 통일** (흐름도의 "포인트" 표기 폐기, UI "크레딧 N개"·"크레딧 내역"과 일치). DB 식별자 `credit`.
   - `CreditLedger` (append-only 거래) + 잔액 캐시. 표시(9-1 크레딧 내역, 커뮤니티 잔액) 전용.
   - 향후 사용처(sink, 예: 기프티콘 교환)가 필요해지면 **자체 통화(옵션 B)로 확장** — 별도 ADR. 원화 실지불 지갑(옵션 A)은 채택 안 함.

6. **유저간 채팅 = Socket.IO + Redis pub/sub** (기존 Redis 재사용).
   - 실시간 1:1 · 그룹(최대 4인). 메시지는 Postgres 영속. 내용 = 텍스트(최대 1000자) + 이미지 첨부 + 스티커/이모지.
   - 오프라인 알림은 기존 `Notification` 모델 재사용.
   - 기존 `ChatSession`/`ChatMessage`(LLM 이벤트 검색용)와는 의미가 달라 **별도 모델**로 분리.

7. **다국어 6종(한/영/베트남/중국/일본/프랑스) = UI 라벨 + 게시글 본문 번역** (사용자 확정).
   - UI: react-i18next 리소스 번들. 게시글 "번역하기": `services/llm` 온디맨드 번역 호출 + 결과 캐시(중복 번역 비용 차단).

8. **캘린더 = 북마크 일정 + 메이트 약속 통합** (사용자 확정).
   - 단일 캘린더에 두 소스 합치고 소스/카테고리로 구분. 와이어의 "카테고리1/2/3"은 placeholder — 실제 라벨은 구현 시 확정.

### 동작 규칙 (와이어프레임·흐름도 확정)

9. **약속 합의**:
   - 1:1 = `같이 가자!` → 약속 팝업(날짜=캘린더, 시간) → 시스템 메시지 버블 → 상대 **수락/거절/날짜·시간 변경(역제안)** 왕복 협상.
   - 그룹 = 참여자 투표 + **전원 동의** 시 확정. 미동의/타임아웃 시 재투표 또는 취소.
   - 확정 시 참여자 캘린더에 일정 등록(GG-ROOM-017/020).
10. **타임아웃 다수 → 백그라운드 스케줄러 도입**: 1:1 신청 24h, 그룹 초대 6h, 강퇴 투표 36h, 약속 동의 36h, 장기 미접속 48h, (Phase 1 기존) 승인 노출 5분, 게시글 만료 7일. 기존 `Notification.scheduledAt` 패턴을 확장.
   - **보정(슬라이스1, 2026-05-30)**: 위 타임아웃 중 **게시글 만료 7일**은 스케줄러 대신 **조회 시점 쿼리 필터(`expires_at > now()`)** 로 구현한다(비노출 + 데이터 보관 GG-POST-010/011/012 을 동일하게 충족, YAGNI). 신청/투표/약속 등 나머지 타임아웃 스케줄러 본체는 후속 슬라이스에서 본 결정대로 도입.
11. **그룹 방장 권한**: 즉시 강퇴 1회(GG-MATE-017), 강퇴 요청→투표(대상 제외 만장일치, 36h, 미응답=동의), 차단/신고, 결원 충원(동의 기반, 약속 확정까지 반복). 최초 신청/개설자 = 방장.
12. **게시판(A_802)**: 카테고리 3종(축제이야기/메이트찾기/자유게시판) + 전체. 댓글/**대댓글 1단계**(대댓글에 답글 불가), **좋아요(하트)**, 본인 글/댓글 수정·삭제, 게시글 7일 만료(비노출, 데이터 보관). *메이트찾기 게시판은 작성자만 댓글자 중 1명에게 1:1 신청 가능(GG-MATE-007/008).*
13. **모더레이션은 관리자 영역 유지** (CLAUDE.md 금지 #4): 신고 접수→관리자 검토→조치(경고/정지/허위신고)는 A_701(이벤트 메이트 처리 페이지). 차단 사용자는 추천/신청/채팅 상호작용에서 제외(GG-REPORT-009).

### 데이터 모델 (Prisma 마이그레이션으로만 — 금지 #2)

14. 신규 모델(개략): `MateProfile`, `MatchRequest`(1:1/그룹), `MateRelation`, `ChatRoom`, `ChatRoomMessage`, `Appointment`(+ 투표/제안), `GroupMembership`(방장/충원/강퇴), `Post`, `Comment`, `PostLike`, `MateEvaluation`, `FestivalSurvey`, `FestivalReview`, `CreditLedger`, `Report`, `Block`. 기존 `Notification`은 메이트 알림 유형 추가로 확장. *상세 스키마는 후속 spec/plan에서 확정.*

### 용어 · 컨벤션 정합

15. **용어 통일**(CLAUDE.md §5-1 연장): `mate`(메이트), `mate_index`(메이트 지수), `credit`(크레딧 — "포인트" 금지), `appointment`(약속), `post`/`comment`(게시글/댓글), `report`/`block`. 기존 `event`·`event_vibe`·`companion_type`·`role{user,uploader,admin}`·`active_role` 불변.
16. **5필터 고정 규칙 불위반**: 메이트 선호조건(성별/연령대/지역/자차/국적/한국어)은 *메이트 매칭* 도메인으로, 이벤트 검색 필터 5종(지역/기간/인원/종류/성향)과 **별개**다. 이벤트 필터엔 추가하지 않는다.
17. **PII**(금지 #3, ADR 0003 정합): MateProfile의 성별·연령대(5세 단위)·지역·국적·한국어 사용여부는 민감정보 → 로그 마스킹, 본인만 접근(GG-NF-MY-002), 개인정보 약관 동의 게이트(GG-MATCH-009, 미동의 시 적용 버튼 비활성).

## Consequences

- (+) 소셜 리텐션·동행 매칭으로 제품 확장.
- (+) 기존 인프라 재사용(Redis pub/sub, OpenAI 번역, Notification, Postgres)으로 신규 인프라 최소화.
- (+) 매칭 룰 기반 → 설명가능·디버깅 용이, LLM 판단 위임 회피.
- (−) **실시간(WebSocket)** + **백그라운드 스케줄러** 신규 운영 표면 추가.
- (−) Prisma 모델 대폭 증가(약 16종) + 마이그레이션 다수.
- (−) PII 수집 표면 확대 — 보안/규제 검토 필요.
- (−) 매칭 품질은 초기 단순(가중치 휴리스틱) — 추후 고도화 여지.
- (−) 게시글 번역 LLM 비용 발생(캐시로 완화).

## Alternatives considered

- **매칭 벡터/임베딩**: 6개 범주형 속성 + 상호 선호 구조에 과하고 불투명. 기각.
- **크레딧 실지불 지갑(옵션 A, PG 연동)**: 명세·NF 어디에도 근거 없음, 전자금융 규제·환불·정산·보안 부담. 현 단계 기각(향후 필요 시 별도 ADR).
- **SSE/폴링 채팅**: 단방향·지연으로 실시간 대화 부적합. 기각.
- **기존 ChatSession/ChatMessage 재사용**: LLM 검색 세션과 의미 충돌. 별도 모델로 분리.
