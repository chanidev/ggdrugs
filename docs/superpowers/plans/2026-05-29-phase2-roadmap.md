# Phase 2 (커뮤니티+메이트) 구현 로드맵

> 근거: ADR 0007 (`docs/decisions/0007-phase2-community-mate-matching.md`), Phase 2 요구사항정의서(2026-04-27, A_800~A_901·A_701, GG-COMM/MATCH/MATE/ROOM/POST/NOTI/PROFILE/REVIEW/FEST-REVIEW/REPORT/NF).
> 대상 액터: **일반 사용자**. 업로더·관리자·이벤트 검색(Phase 1)은 불변. 진입점: 마이페이지(A_500) → 커뮤니티(A_800).
> 본 문서는 슬라이스 분할·의존·빌드순서·리스크만 다룬다. 슬라이스별 상세 task/스키마는 별도 plan에서 확정한다.

## 1. 빌드 순서 한눈에

```
의존 체인:  1 → 2 → 3 → 4 → 5  →  (6 · 8 마무리)
횡단 슬라이스: 7(i18n) · 9(스케줄러)  →  1·3에서 스캐폴드, 이후 정착
```

- **1→2→3→4→5** 직선 의존(커뮤니티 셸 → 프로필/추천 → 신청·채팅 → 약속 → 평가·크레딧).
- **6(알림)·8(신고/차단)**: 앞 슬라이스의 이벤트 소스가 갖춰진 뒤 통합 마무리.
- **7(i18n)·9(스케줄러)**: 횡단 관심사. 슬라이스 1(i18n 스캐폴드)·3(스케줄러 스캐폴드)에서 골격을 세우고, 7은 슬라이스 5 이후 본문 번역까지 확장, 9는 타임아웃 잡이 누적된 뒤 정착시킨다.

| 예상 위험 | 실시간(WebSocket)과 함께 신규 운영 표면 → **격리·멱등성·재시작 복원** 필수. 중복 실행/누락, 멀티 인스턴스 시 잡 단일 실행 보장. BFF REST 응답성에 영향 차단 |

---

## 3. 횡단 리스크 요약

- **실시간·스케줄러 격리**: 슬라이스 3·9의 WebSocket·워커는 BFF REST 가용성과 분리된 경계로 운영. 장애가 핵심 검색/게시판으로 전파되지 않도록 한다.
- **채팅 모델 혼동 금지**: `ChatRoom`/`ChatRoomMessage` ≠ `ChatSession`/`ChatMessage`(LLM 검색). 절대 재사용 안 함.
- **크레딧 옵션 C YAGNI**: 적립·표시만. sink·결제·멤버십·구매·쿠폰은 placeholder로 두고 구현하지 않음(확장 시 별도 ADR).
- **PII 표면 확대**: MateProfile 민감정보 마스킹·본인접근·동의 게이트(금지 #3, ADR 0003).
- **LLM 판단 위임 금지**: 매칭·메이트 지수·모더레이션 모두 결정적/관리자 영역. LLM은 게시글 번역에만(금지 #4).
- **ADR Open items(미확정 상수)**: 매칭 가중치, 메이트 지수 산식 상수, 캘린더 카테고리 라벨, 좋아요(하트 PDF 미기재·와이어 채택) → 코드에 하드코딩하지 말고 설정값으로 외부화하여 슬라이스 plan에서 확정.
