# Ⅰ. 개요

## 1. 분석 목적

본 문서는 **Alle** 서비스의 데이터 수집·정제·분석·모델링 전 과정을 정의한다. Alle는 전국에 분산된 공공 이벤트 정보를 단일 스키마로 통합하고, 이를 **AI 요약·임베딩·뉴스 매핑**을 거쳐 가공한 뒤, **지도 + 자연어 채팅 UI** 위에서 검색·추천·구독하도록 하는 서비스다. 나아가 같은 행사를 함께 갈 **동행(메이트)을 규칙 기반으로 매칭**한다.

빅데이터 분석 관점에서 본 정의서의 목적은 다음과 같다.

1. **데이터 정의** — 어떤 외부 데이터원으로부터 어떤 형식의 데이터를 수집·축적하는지 명세한다.
2. **수집·정제 방법 정의** — 분산·이질적 소스를 단일 스키마로 정규화하는 24시간 주기 파이프라인을 정의한다.
3. **분석·모델링 정의** — 사전학습 임베딩 모델 + 벡터 검색(Qdrant) + LLM 요약/추천으로 구성된 **임베딩 기반 RAG** 구조를 정직하게 서술한다. (Alle는 LLM 파인튜닝을 수행하지 않는다.)

## 2. 분석 배경

### 2-1. 문제 의식

국내 이벤트·행사 정보는 **공급원이 극도로 분산**되어 있다. 동일 행사 정보가 한국관광공사 TourAPI, 서울 열린데이터광장(Seoul Open Data), 한국문화정보원(KCISA) 등 복수 공공 API에 산재하며, 각 소스의 **분류 체계·지역 표기·기간 표기가 서로 달라** 사용자가 한곳에서 통합 탐색하기 어렵다. 또한 기존 검색은 **키워드 매칭 중심**이어서 "이번 주말 연인이랑 강남에서 볼만한 공연" 같은 **자연어 의도**를 그대로 받아내지 못한다.

### 2-2. 세 가지 미충족 니즈

| 니즈 | 현황의 한계 | Alle의 접근 |
|---|---|---|
| **자연어 이벤트 검색** | 키워드·필터 조합을 사용자가 직접 구성해야 함 | 대화형 채팅으로 의도를 5종 필터 + 의미 검색으로 자동 변환 |
| **분산된 행사 정보 통합** | 소스마다 포맷·지역·분류가 상이 | 3개 공공 API를 단일 스키마로 정규화·중복제거·지오코딩 |
| **동행(메이트) 매칭 니즈** | 같은 행사를 함께 갈 동행을 찾을 채널 부재 | 같은 이벤트 선택자 풀에서 규칙 기반 양방향 매칭 |

### 2-3. 데이터의 시의성·이슈 연계

이벤트는 **시점성**이 강하다. 종료된 행사를 추천하면 가치가 없으므로 `phase`(upcoming / ongoing / ended) 상태머신과 `period`(3m / 6m / all / custom) 기간 필터로 시점을 통제한다. 나아가 각 이벤트를 **뉴스 기사와 매핑**(이벤트·이슈 연계)하여 사용자가 행사의 맥락·화제성까지 함께 파악하도록 한다. 현재 approved 이벤트 **4,111건** 중 **1,810건(44%)** 에 뉴스 매핑이 연결되어 있다.

# Ⅱ. 기능적·빅데이터 분석 명세

## 1. 데이터 정의

### 1-1. 수집 데이터원 (Sources)

Alle의 빅데이터는 **외부 공공 데이터(이벤트)**, **외부 뉴스 데이터(이슈)**, **서비스 내부 생성 데이터(사용자 행동·취향)** 의 세 계층으로 구성된다.

#### (1) 외부 이벤트 데이터원

| 소스 | 수집 대상 | 비중 / 역할 |
|---|---|---|
| **Seoul Open Data**(서울 열린데이터광장) | 서울 문화행사 8개 카테고리 | 압도적 다수 |
| **TourAPI**(한국관광공사) | 전국 축제(forward-looking) | 전국 확장 핵심 |
| **KCISA**(한국문화정보원) | 공연·전시 | 키 의존(보유 상태에 따라 변동) |

#### (2) 외부 뉴스 데이터원 (이벤트-이슈 연계)

| 소스 | 수집 대상 | 역할 |
|---|---|---|
| **Naver 뉴스 검색 API** | 이벤트 키워드 관련 기사 | 이벤트-뉴스 매핑 1차 |
| **Google News RSS** | 이벤트 키워드 관련 기사 | 보강 소스 |

#### (3) 내부 생성 데이터원

| 분류 | 데이터 | 적재 테이블 |
|---|---|---|
| 검색·행동 로그 | 필터/채팅 검색 파라미터, LLM 대화 코퍼스 | `search_logs`, `chat_sessions`, `chat_messages` |
| 취향·개인화 | 선호 차원/값(EAV), 사진 AI 태그 | `user_taste_profiles`, `photos.ai_tags` |
| 사용자 콘텐츠 | 북마크, 리뷰(+감성), 구독 조건 | `bookmarks`, `reviews`, `event_subscriptions` |

### 1-2. 데이터 형식 (Schema / Format)

저장소는 **PostgreSQL + PostGIS**(extensions: postgis, pg_trgm, unaccent, citext)를 중심으로, 벡터는 **Qdrant v1.13**(스키마 외부), 오브젝트는 **MinIO 4버킷**에 분산 저장한다. 전체 데이터 모델은 **43개 테이블**로 구성된다.

#### (1) 핵심 적재 테이블

| 테이블 | 형식 핵심 컬럼 | 빅데이터 관점 의미 |
|---|---|---|
| **events** | `title VARCHAR(200)`, `description TEXT`, `ai_summary TEXT`, `location_geom geometry(Point,4326)`, `start_date/end_date DATE`, `approval_status`, `phase` | 모든 행사의 상위 개념(festival로 대체 금지). 지오·기간·상태가 분석의 기준 축 |
| **news_articles** | `original_url VARCHAR(500) U`, `content_body/summary TEXT`, `metadata JSONB`, `published_at` | 외부 뉴스 원천. 이벤트-이슈 매핑의 좌측 입력 |
| **event_article_mappings** | `event_id`, `article_id`, `relevance_score NUMERIC(5,4)`, `matched_at` | 뉴스 매핑 결과(커버리지 44%) 적재처 |
| **search_logs** | `search_type(filter/chat)`, `search_params JSONB` | 검색 행동 코퍼스 |
| **user_taste_profiles** | `taste_dimension`, `taste_value` (EAV) | 개인화 추천 입력 |
| **reviews** | `body TEXT`, `rating SMALLINT`, `sentiment` | 감성분석 산출 적재 |

#### (2) Prisma → SQL 형식 매핑(발췌)

| Prisma 타입 | PostgreSQL 형식 |
|---|---|
| `String` / `@db.Text` | VARCHAR / TEXT |
| `@db.Uuid` / `@db.Char(n)` | UUID / CHAR(n) |
| `Int` / `@db.SmallInt` | INTEGER / SMALLINT |
| `@db.Decimal(p,s)` | NUMERIC(p,s) |
| `@db.JsonB` | JSONB |
| `Unsupported("geometry(Point,4326)")` | geometry(Point,4326) — WGS84 |

#### (3) 벡터 형식

이벤트 텍스트는 **text-embedding-3-small (1536차원)** 으로 임베딩되어 Qdrant `alle-events` 컬렉션에 **cosine 거리** 포인트로 저장된다. PostgreSQL 스키마에는 벡터 컬럼이 존재하지 않으며, 벡터는 Qdrant가 전담한다.

| phase 분포 | upcoming 163 / ongoing 260 / ended 3,661 |

## 2. 데이터 수집 방법

### 2-1. 수집 오케스트레이션 (24h 주기 scheduler)

BFF(Node 22 + Express 5)의 ingest scheduler가 **부팅 +2초** 후 **24시간 주기**로 3개 러너(TourAPI / Seoul / KCISA)를 병렬 수집한 뒤, 공통 정제 8단계와 후속 직렬 4단계를 실행한다.

```
부팅+2s, 24h 주기 — 3 러너 병렬 수집(TourAPI / Seoul / KCISA)
└ ingest-common 8단계
   1) 지역 guard          2) 시군구 추출        3) regionId resolve
   4) 카테고리 8종 매핑     5) cross-source 중복제거(title+start_date)
   6) phase 계산          7) approval_status 부여   8) Prisma upsert
└ 후속 직렬 4단계
   1) AI 요약(gpt-4o-mini)   2) 뉴스 매핑(Naver+Google)
   3) 임베딩 → Qdrant upsert  4) 매핑 분포 감사
```

### 2-2. 정제 단계 상세

| 단계 | 처리 내용 |
|---|---|
| **지역 정규화** | `(sido, sigungu)` exact → 시 단위 → 광역 → null 순의 fallback resolver로 `regions` 테이블에 매핑. 실측에서 TourAPI 표본 393건 전수 매칭(0 errors) 확인 |
| **카테고리 매핑** | 소스별 분류를 `event_categories` 마스터 8종 코드(festival / expo / symposium / conference 등)로 정규화 |
| **중복 제거** | `title + start_date` 키로 cross-source 중복 제거(동일 행사가 복수 API에 존재) |
| **phase 계산** | `start_date / end_date`와 현재일을 비교하여 upcoming / ongoing / ended 산정 |
| **승인 상태 부여** | 크롤링 데이터에 `approval_status` 부여(상태머신 진입) |
| **upsert** | `crawl_origin + external_source_id` 유니크 쌍 기준 Prisma upsert(멱등) |

### 2-3. 지역 확장

초기 서울 전용에서 **전국 17개 시/도 + 약 230개 시/군/구**로 지역 범위를 확장하였다. `regions` 테이블은 `sido / sigungu / dong`과 `center_lat / center_lng NUMERIC(10,7)`를 보유하여 거리 정렬의 anchor로 사용된다.

## 3. 데이터 분석 결과

### 3-1. AI 분석·가공 작업

수집된 원천 데이터는 후속 4단계에서 다음과 같이 분석·가공된다.

| 분석 작업 | 방법 | 산출 |
|---|---|---|
| **이벤트 요약** | gpt-4o-mini — "신문 도입부 톤, 2~3문장, 250자, 사실만" 프롬프트. `description_hash CHAR(32)` 변경 시 무효화 | `events.ai_summary` |
| **임베딩** | text-embedding-3-small (1536d) → Qdrant `alle-events` upsert | Qdrant 포인트 |
| **뉴스 매핑 점수** | `final = 0.4 × keyword_overlap + 0.6 × embedding_cosine`(임계 0.60 / 0.55) | `event_article_mappings.relevance_score NUMERIC(5,4)` |
| **리뷰 감성분석** | gpt-4o-mini 분류(positive / negative / neutral), 키 없으면 사전 기반 fallback | `reviews.sentiment` |
| **사진 태깅** | AI 태깅 | `photos.ai_tags (JSONB)` |

### 3-2. 뉴스 매핑 분석 결과 (이벤트-이슈 연계)

`news_articles`(네이버 뉴스 검색 API + Google News RSS 수집, `original_url` 유니크)를 좌측 입력으로, 이벤트 텍스트와의 **키워드 중복(0.4) + 임베딩 코사인(0.6)** 가중합으로 관련도를 산출하여 임계 이상 기사만 `event_article_mappings`에 적재한다. 그 결과 approved 4,111건 중 **1,810건(44%)** 에 관련 뉴스가 연결되었다. 매핑되지 않은 56%는 화제성이 낮거나 최근 보도가 없는 이벤트로, 후속 회차에서 재시도된다.

### 3-3. 검색·행동 로그 분석

| 분석 대상 | 데이터 | 활용 |
|---|---|---|
| 검색 패턴 | `search_logs`(filter / chat, JSONB params) | 필터 사용 빈도·자연어 의도 분포 분석 |
| 대화 코퍼스 | `chat_sessions` / `chat_messages`(user / assistant) | LLM 검색 품질 모니터링 |
| 콘텐츠 반응 | `bookmarks`, `reviews.sentiment` | 인기·만족도 신호 |

> 위 항목은 분석 **설계·수집 경로**를 정의한 것으로, 운영 초기 수집 단계이므로 실제 집계 수치는 산출되지 않았다 — 집계 결과는 후속 리포트에서 제시 예정 **(TBD)**. 검색 로그 파티셔닝 또한 Phase 3+ 로 연기되었다 **(TBD)**.

### 3-4. 취향 프로필 분석

`user_taste_profiles`는 **EAV(Entity-Attribute-Value)** 구조로 사용자별 선호 차원(`taste_dimension`)과 값(`taste_value`)을 축적한다. 추천 단계에서 선호 카테고리 / 지역 / 성향 3차원과 seed 이벤트의 평균 벡터(mean-vector) kNN을 결합하여 개인화 추천을 생성한다.

> 취향 차원별 분포·선호 신호 집계는 **운영 초기 수집 단계**로, 의미 있는 통계 산출 전이다 — 집계 결과는 후속 리포트에서 제시 예정 **(TBD)**.

### 3-5. 거버넌스 — LLM 판단 위임 금지

LLM은 **추천·검색·요약 전용**이다. 이벤트 승인(approval), event_vibe 라벨 부여, 업로더 승급 심사, 신고 모더레이션은 모두 **관리자(사람)의 결정 영역**이며 LLM에 위임하지 않는다. 이는 설명가능성·감사가능성 확보를 위한 의도적 설계다.

# Ⅲ. 모델링 및 학습

> **핵심 전제**: Alle는 자체 LLM **파인튜닝(fine-tuning)을 수행하지 않는다.** 따라서 본 장은 "모델 학습"이 아니라 **사전학습 임베딩 모델 + 벡터 검색(Qdrant) + LLM 요약/추천**으로 구성된 **임베딩 기반 RAG(Retrieval-Augmented Generation)** 파이프라인으로 정직하게 서술한다.

## 1. 모델 방법

### 1-1. 모델 구성

| 구성 요소 | 모델 / 기술 | 역할 |
|---|---|---|
| **임베딩 모델** | text-embedding-3-small (1536d) | 이벤트·질의 텍스트를 벡터로 인코딩 |
| **벡터 검색** | Qdrant v1.13 (`alle-events`, cosine) | 의미 기반 후보 검색(RAG의 Retrieval) |
| **채팅 LLM** | gpt-4o-mini (`OPENAI_MODEL_FAST`) | 자연어 의도 → 필터 구조화, 후보 재정렬·요약(RAG의 Generation) ¹ |
| **보조 LLM** | gpt-4o-mini (`OPENAI_MODEL_FAST`) | 이벤트 요약, 분류, 감성분석, 번역 |
| **구조화 출력** | structured outputs (strict JSON) | 자연어 → 필터 5종 추출의 스키마 강제 |
| **규칙 기반 fallback** | 사전 기반 매처 | 키 미설정·예산 초과 시 조용히 대체 |

> **단일 모델 사용**: 자연어→필터 추출, reply 생성, 후보 재정렬, 이벤트 요약, 리뷰 감성분석, 번역이 **전부 gpt-4o-mini**(`OPENAI_MODEL_FAST` 기본값)로 처리된다. 실제 구현상 gpt-4o를 직접 호출하지 않는다.
>
> ¹ 고품질 모델 전환은 환경변수(`OPENAI_MODEL_FAST`)로 모델을 전환하여 가능하며, gpt-4o는 실제 호출 모델이 아니라 **전환 옵션 표기**다.

> **LangChain 미사용**: 본 서비스는 LangChain 라이브러리에 의존하지 않는다. 임베딩·벡터검색·LLM 호출은 모두 **OpenAI SDK 직접 구현** + structured outputs 방식이다.

### 1-2. 자연어 채팅 검색 모델 흐름 (/chat/stream — 5 step)

```
1) 개인화 컨텍스트 구성   → taste profile + 최근 북마크
2) LLM 필터 추출         → 필터 5종(지역/기간/인원구성/종류/성향) + reply + followups (structured output)
3) ID resolve           → 지역·성향을 마스터 테이블 ID로 변환
4) Qdrant 의미 검색      → over-fetch 30, score ≥ 0.25
5) LLM 재정렬            → 후보 ≥ 6 & 8자↑ 시 top 12 → top 5 + matchReason(1줄)
   (0건 시 검색 무응답(결과 0건) 안내)
```

이 흐름은 전형적 RAG다. 사용자 발화를 임베딩 검색으로 **회수(Retrieve)** 한 뒤, 회수된 후보만을 컨텍스트로 LLM이 **재정렬·근거 생성(Generate)** 한다. LLM은 후보 풀 밖의 이벤트를 창작하지 않는다.

### 1-3. 추천 모델 (/me/recommendations)

`user_taste_profiles`의 3차원(선호 카테고리 / 지역 / 성향)과 seed 이벤트의 **평균 벡터(mean-vector) 기반 kNN**으로 추천을 산출한다.

### 1-4. 매칭 모델 (Phase 2 — 의도적 비-LLM)

메이트 동행 매칭은 **벡터/임베딩이 아닌 규칙 기반 양방향 가중 점수**로 결정적으로 계산한다.

```
① 하드 필터  : selected_event_id 동일 풀 한정 + 선호 불일치 제외
② 소프트 점수 : 지역 근접 · 연령대 차 · 국적/한국어/자차 가중 합산
③ 동점 처리  : 메이트 지수(평가 기반 집계, 사용자 수정 불가)로 정렬
```

설명가능성·디버깅 용이성을 위해 매칭에서 LLM을 의도적으로 배제한다.

## 2. 학습 방법

### 2-1. 파인튜닝 미수행 — 사전학습 + RAG 채택 근거

Alle는 가중치 갱신(weight update)이나 자체 모델 학습을 수행하지 않는다. 그 대신 다음을 채택한다.

| 항목 | 채택 방식 | 사유 |
|---|---|---|
| 도메인 적응 | **임베딩 기반 RAG** (검색으로 최신 이벤트 주입) | 이벤트는 24h마다 갱신 — 파인튜닝은 최신성 확보 불가 |
| 출력 제어 | **structured outputs(strict JSON)** | 파인튜닝 없이도 필터 5종 스키마 강제 가능 |
| 비용·운영 | **사전학습 모델 + 규칙 fallback** | 학습 비용·MLOps 부담 제거, 예산 초과 시 graceful degradation |

### 2-2. "학습" 대체 절차 — 임베딩 인덱싱 루프

파인튜닝의 학습 루프에 대응하는 것은 **임베딩 인덱싱 루프**다. 24시간 주기 수집의 후속 3단계에서 신규·변경 이벤트만 재임베딩하여 Qdrant에 upsert한다. `description_hash` 변경 시에만 요약·임베딩을 무효화·재생성하므로, 변경분만 갱신하는 **증분 인덱싱(incremental indexing)** 으로 동작한다.

```
신규/변경 이벤트 감지(description_hash diff)
   → AI 요약 재생성(gpt-4o-mini)
   → 임베딩 재계산(text-embedding-3-small)
   → Qdrant alle-events upsert
   → 매핑 분포 감사(coverage 모니터링)
```
